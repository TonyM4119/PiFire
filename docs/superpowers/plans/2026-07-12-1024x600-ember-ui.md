# 1024×600 ember dashboard variant — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a 1024×600 resolution variant of the ember dashboard to both the pygame/DSI display and the Qt Quick (QML) display, using a native reflow (not a uniform down-scale).

**Architecture:** The pygame side is data-driven — a bespoke `_dashboard_1024x600()` layout is added to `tools/generate_dsi_layout.py`, which emits `display/dsi_1024x600t.json`; the widgets already resize their rendered image to the JSON box, so no widget code changes. The Qt side is fixed-pixel QML — a `compact` flag threaded through `DashScreen.qml` and its components switches every pinned dimension to a smaller value at 1024×600 while leaving 1280×720 byte-for-byte unchanged.

**Tech Stack:** Python 3, Pillow (PIL) for pygame rendering, pygame-ce; PySide6 / Qt Quick (QML) for the Qt UI; pytest.

## Global Constraints

- Existing resolutions must not regress: `dsi_800x480t`, `dsi_1024x768t`, `dsi_1280x720t` JSON output stays byte-for-byte identical; the Qt `compact` flag is `false` by default so 1280×720 QML renders unchanged.
- pygame widgets are NOT modified — every ember widget in `display/flexobject.py` already renders to a fixed working canvas and calls `.resize(output_size)`, so it scales to whatever box the layout JSON gives.
- New module files are thin re-export shims (`from display.dsi_800x480t import Display` / `from display.qtquick_flex import Display`), matching `dsi_1280x720t.py` and `qtquick_dsi_1280x720t.py`.
- Wizard manifest default for the new pygame module: `display_data_filename` = `./display/dsi_1024x600t.json`; for Qt: `./display/qtquick_dsi_1024x600t.json`.
- `accent_theme` config option present on both new manifest entries, default `Ember`, list `{Ember, Ice, Crimson}`.
- Minimum QML font `pixelSize` is 11 — never scale a font below it.
- PySide6/Qt is in **system python** (`/usr/bin/python3`), not the `.venv`. Run Qt tests with `QT_QPA_PLATFORM=offscreen`.
- Run `uvx ruff format` on changed Python files before every commit.
- The authoritative 1024×600 layout is the coordinate table below; both back-ends implement it.

The radial-gradient stops and the `0.50 / 1.18` center and `1.20 / 0.90` radius fractions are resolution-independent — keep them, just compute against each `(W, H)`.

```python
"""Render the ember radial-gradient dashboard background for each DSI resolution."""

import os
from PIL import Image

# (width, height) of every ember dashboard background this tool owns.
RESOLUTIONS = [(1280, 720), (1024, 600)]

# radial-gradient(120% 90% at 50% 118%, #241a12 0%, #16110d 42%, #0d0b09 100%)
STOPS = [(0.00, (0x24, 0x1A, 0x12)), (0.42, (0x16, 0x11, 0x0D)), (1.00, (0x0D, 0x0B, 0x09))]


def _lerp(a, b, t):
	return tuple(round(a[i] + (b[i] - a[i]) * t) for i in range(3))


def _sample(frac):
	for i in range(len(STOPS) - 1):
		p0, c0 = STOPS[i]
		p1, c1 = STOPS[i + 1]
		if frac <= p1:
			t = 0 if p1 == p0 else (frac - p0) / (p1 - p0)
			return _lerp(c0, c1, t)
	return STOPS[-1][1]


def render(width, height):
	cx, cy = 0.50 * width, 1.18 * height
	rx, ry = 1.20 * width, 0.90 * height
	img = Image.new('RGB', (width, height))
	px = img.load()
	for y in range(height):
		for x in range(width):
			d = (((x - cx) / rx) ** 2 + ((y - cy) / ry) ** 2) ** 0.5
			px[x, y] = _sample(min(d, 1.0))
	return img


def out_path(width, height):
	base = os.path.abspath(os.path.join(os.path.dirname(__file__), '..'))
	return os.path.join(base, 'static', 'img', 'display', f'background_ember_{width}x{height}.png')


def main():
	for width, height in RESOLUTIONS:
		path = out_path(width, height)
		render(width, height).save(path)
		print(f'Wrote {path}')


if __name__ == '__main__':
	main()
```
