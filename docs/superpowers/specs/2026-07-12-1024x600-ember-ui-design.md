# 1024×600 ember dashboard variant — design

Date: 2026-07-12

## Goal

Add a 1024×600 resolution variant of the new "ember" dashboard, for **both** display
back-ends that render it:

- the pygame / DSI flex display (`display/base_flex.py` + `display/flexobject.py`, data
  from generated JSON), and
- the Qt Quick / QML display (`display/qtapp.py` + `display/qml/`).

The ember dashboard currently exists only at 1280×720 (`dsi_1280x720t` and
`qtquick_dsi_1280x720t`). There is **no** 1280×800 variant — 1280×720 is the newest UI and
the reference design for this work.

## Approach

**Native reflow**, not uniform scaling. Coordinates are recomputed for 1024×600 so the
dashboard fills the screen with readable text, rather than scaling the 1280×720 layout by
0.8× (which would shrink text ~20% and letterbox the slightly-wider aspect ratio). This
matches the existing pattern: the 1280×720 ember dash is a bespoke hand-authored layout
(`_dashboard_1280x720()`), not an auto-scale of the 800×480 source.

The two back-ends share one visual design; only the rendering mechanism differs.

## Layout (both back-ends)

Screen 1024×600. Margins 14px. Three columns; content bottoms at y=574 (12px bottom margin).

| Region | 1280×720 (reference) | 1024×600 (new) |
|---|---|---|
| Header bar | `[0,0]` `1280×58` | `[0,0]` `1024×50` |
| Left: 5 probe cards | x=18, `298×116`, y=74/202/330/458/586 | x=14, `238×96`, y=62/166/270/374/478 |
| Center gauge (`gauge_ember`) | `[332,74]` `614×452` | `[266,62]` `490×352` |
| Cook-time pill | below gauge | `[266,424]` `490×42` |
| Control / button row | y=602, `×100` | `[266,476]` `490×98` |
| Right: system card | `[962,74]` `300×230` | `[770,62]` `240×190` |
| Right: 2 duty pills | x=962 | `[770,262]` / `[770,310]`, `240×40` |
| Right: hopper (vertical) | `300×306` | `[770,358]` `240×216` |

Column arithmetic (must hold exactly):
`14 + 238 + 14 + 490 + 14 + 240 + 14 = 1024`.

Row arithmetic (left column): `62 + 5×96 + 4×8 (gaps) = 62 + 480 + 32 = 574`.

`profile_2` (rotated / portrait) is produced by the same generator machinery that already
handles rotation for the other resolutions; only `profile_1` (landscape) is hand-tuned.

## Widget font scaling — no shared-widget change needed (pygame)

Original assumption (during brainstorming) was that `ProbeCard`, `SystemCard`, `DutyPill`,
and `HeaderBar` used hardcoded point sizes that would overflow the smaller 1024×600 cards,
requiring a box-relative font refactor. **Reading the code disproved this.** Every ember
widget in `display/flexobject.py` renders to a fixed high-resolution "working canvas" with
fixed fonts and then resizes the finished image to the JSON-configured box:

```python
size = (400, 220)                 # fixed working canvas
# ... draw name/temp/units at fixed point sizes ...
card = card.resize(output_size)   # scales the whole image to the layout's [w, h]
```

Confirmed present in ProbeCard, SystemCard, DutyPill, HeaderBar, GaugeEmber,
HopperVertical, ButtonRow, AlertMessage, ModeBar, ControlPanel (each has a
`.resize(output_size)` tail). So a `ProbeCard` at `[238, 96]` already shrinks its text to
fit. **No shared-widget change is required for the pygame variant**, and the existing
resolutions carry no regression risk from this work.

The pygame side therefore reduces to: a bespoke `_dashboard_1024x600()` layout in the
generator, the module/JSON/manifest wiring, and tests.

## Qt Quick reflow — the real layout work lives here

QML uses *live* fixed pixel sizes with no working-canvas resize: `Gauge` font
`pixelSize: 84`, `HeaderBar height: 58`, column widths `298`/`300`, center
`minimumWidth: 380`, gauge box `392×392`, gauge-card `minimumHeight: 420`, the `52`/`82`
fixed rows, and the `64` duty-pill row. Shrinking the window to 1024×600 overflows both
axes, so the Qt UI needs a genuine reflow.

Approach: a `compact` flag driven off the actual dash width — `DashScreen.qml` gets
`readonly property bool compact: width <= 1100`, and every pinned dimension becomes
`compact ? <1024x600 value> : <1280x720 value>`. Components with intrinsic fixed sizes
(`HeaderBar`, `Gauge`, `CookTimeBar`, `ControlPanel`, `DutyPill`, `SystemCard`,
`HopperCard`, `ProbeCard`) each gain a `compact` boolean property (default `false`,
preserving current 1280×720 rendering) that switches their internal sizes/font pixelSizes.
`DashScreen.qml` passes `compact: dash.compact` down to each. At 1280×720 every value is
unchanged, so that resolution is untouched.

1024×600 target values (mirroring the pygame layout table):
- header height 58 → 50
- body margins 18/18/16/18 → 14/14/12/14, spacing 16 → 14
- left column width 298 → 238
- center `minimumWidth` 380 → 300; gauge box 392 → 300; gauge-card `minimumHeight` 420 → 300
- cook-time row 52 → 42; control-panel row 82 → 74
- right column width 300 → 240; duty-pill row 64 → 40; hopper `minimumHeight` 180 → 140
- component font pixelSizes reduced ~0.8× where they would otherwise clip
  (e.g. Gauge temp 84 → 66, °units 30 → 24; HeaderBar wordmark 20 → 16)

## Files

### pygame / DSI
- `tools/generate_dsi_layout.py` — add `(1024,600)` to `RESOLUTIONS`; add
  `_dashboard_1024x600()`; add a `(1024,600)` branch in `build()`; regenerate.
- `display/dsi_1024x600t.py` — new re-export shim: `from display.dsi_800x480t import Display`.
- `display/dsi_1024x600t.json` — generated output (do not hand-edit).
- `wizard/wizard_manifest.json` — new `dsi_1024x600t` module entry (clone `dsi_1024x768t`).
- (No `display/flexobject.py` change — widgets already resize to their box.)
- `static/img/display/background_ember_1024x600.png` — ember background sized 1024×600
  (optional splash asset; splash currently shared).

### Qt Quick
- `display/qtquick_dsi_1024x600t.py` — new shim: `from display.qtquick_flex import Display`.
- `display/qtquick_dsi_1024x600t.json` — metadata `screen_width:1024`, `screen_height:600`.
- `display/qml/screens/DashScreen.qml` — add `compact` flag; make each pinned dimension
  `compact ? small : large`; pass `compact` down to components.
- `display/qml/components/*.qml` — HeaderBar, Gauge, CookTimeBar, ControlPanel, DutyPill,
  SystemCard, HopperCard, ProbeCard each gain a `compact` bool (default false) switching
  internal sizes/font pixelSizes.
- `wizard/wizard_manifest.json` — new `qtquick_dsi_1024x600t` module entry.

### Tests
- `tests/test_dsi_layout_generator.py` — byte-identical regression assertion for the new JSON.
- `tests/test_dsi_1024x600t_{module,layout,manifest}.py` — mirror the 1280×720 suites.
- `tests/test_qtquick_manifest.py`, `tests/test_qtquick_display.py`, `tests/test_qml_load.py`,
  `tests/test_qtquick_parity.py` — extend / parallel for the 1024×600 Qt variant.

## Verification

After implementation, run each UI at 1024×600 offscreen and capture **real screenshots**
of both the pygame and Qt Quick dashboards for sign-off — the mockup is not sufficient
evidence. Also run the full test suite. The Qt reflow uses a `compact` flag that is off at
1280×720, so that resolution must be re-verified as visually unchanged. PySide6/Qt is in
system python (`/usr/bin/python3`), not the `.venv`.

## Out of scope

- No 1280×800 variant (does not exist; not requested).
- No changes to the legacy fixed-size pygame modules (`pygame_240x320`, SPI panel drivers).
- No new dashboard *features* — this is a resolution port of the existing ember design.
